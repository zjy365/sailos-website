<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repository source maps -> audit report | Planning and codebase-map files are trusted inputs for the ledger, while repo file inventory can change during execution. |
| Audit report -> later phase executors | Later workers rely on the report for schema, duplicate keys, and module boundaries. |
| Local environment -> audit evidence | Validation commands read local files; secret-bearing `.env*` files stay outside Phase 1 evidence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `docs/performance-audit.md` schema and duplicate keys | mitigate | Validate required headings, schema fields, status values, validation values, and duplicate-key tokens with the automated Node checks in this plan. |
| T-01-02 | Repudiation | Module inventory provenance | mitigate | Cite `.planning/codebase/*`, `.planning/research/*`, and repo surface paths in the report sections so later findings can trace their source. |
| T-01-03 | Information Disclosure | Audit evidence collection | mitigate | Record file paths and source-level evidence only; keep `.env*` files and secret values out of the report. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | This plan has zero package-manager install tasks; any added install step must stop for a package legitimacy audit before execution. |
</threat_model>
